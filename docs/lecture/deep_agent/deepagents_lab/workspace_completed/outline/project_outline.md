# OpenClaw Secretary Agent for Office Productivity

## Executive summary and context

This project proposes an OpenClaw-based secretary agent that can help a user manage four common office workflows that are usually scattered across separate tools: inbox triage, calendar coordination, meeting preparation, and follow-up task tracking. The core motivation is that these workflows are not independent. A useful assistant must read an incoming email, infer whether it requires a reply or a meeting, inspect the calendar, prepare context for the upcoming conversation, and convert commitments into explicit tasks. Recent agent benchmarks suggest that these cross-application workflows remain difficult even when the environment is controlled. WorkBench introduces a realistic workplace benchmark with multi-step tasks across email, calendar, analytics, and project-management tools, and evaluates agents by whether they reach the correct end state rather than whether they copy a gold action trace. OfficeBench similarly focuses on office automation across documents, email, and calendars, while OSWorld shows that open-ended computer-use tasks remain challenging even for strong multimodal agents. Taken together, these systems motivate a project that is narrower than “general office automation” but more realistic than a single-task email or scheduling bot.

The proposed problem is to build and evaluate a constrained secretary agent inside OpenClaw that can operate over realistic office workflows with measurable state changes and explicit safety checks. OpenClaw is a suitable substrate because its official repository and documentation position it as a personal assistant platform with multi-agent routing, persistent workspace context, and messaging-based operation. That makes it a plausible environment for hosting a planner plus several role-specific agents while keeping the user-facing experience unified. The research question for the project is whether a multi-agent OpenClaw secretary can outperform simpler baselines on realistic office tasks while keeping critical mistakes and user oversight burden at acceptable levels.

## Problem and motivation

The problem this project is solving is not simply email classification or meeting summarization in isolation. It is the broader problem of coordinated office assistance under real constraints. In everyday work, users lose time because inboxes, calendars, meeting notes, and task trackers are fragmented. A request to “set up a meeting with the data team next week” may require understanding urgency from email context, checking multiple calendars, finding feasible time slots across time zones, drafting a reply, and then tracking action items after the meeting. Current language agents often perform well on single prompts but remain brittle on workflows that require tool use, persistent state, and cross-application reasoning. Secretary work is also high stakes: a wrong-recipient email, an incorrect meeting time, or a missing follow-up task can impose real costs.

This project therefore frames the secretary agent as a realistic workflow problem with both effectiveness and safety requirements. The system will target four bounded capabilities. First, it will triage incoming email by identifying whether a message is informational, urgent, scheduling-related, or action-item-bearing. Second, it will support calendar coordination by checking constraints and proposing or creating events. Third, it will generate meeting preparation briefs that summarize relevant context, open questions, and dependencies from recent threads and prior meeting notes. Fourth, it will track follow-up tasks by extracting commitments from meetings and email threads and writing them into an explicit task backend. These four functions are manageable for a class project, but together they capture the core value proposition of a secretary-style agent.

## What the team wants to build

The team plans to build a constrained multi-agent assistant on top of OpenClaw rather than a single monolithic agent. The system will expose one user-facing secretary persona, but internally it will consist of specialized agents coordinated by a supervisor. The supervisor agent will interpret user requests or proactive triggers, decompose work into subgoals, route those subgoals to specialized workers, and enforce guardrails. It will be the only component allowed to authorize externally visible actions such as sending a draft or modifying a meeting, and even then high-risk actions will require user confirmation. This design is motivated both by OpenClaw’s support for multi-agent routing and by evidence from recent office-agent work suggesting that long, cross-application tasks benefit from explicit task decomposition.

Four specialist agents will handle the main workflow. An inbox-triage agent will read new email, classify urgency and intent, retrieve relevant thread or calendar context, and recommend one of a small number of actions such as archive, defer, draft reply, scheduling follow-up, or create task. A calendar-coordination agent will inspect availability, hard constraints, time zones, attendee requirements, and meeting priority before proposing one or more feasible options or creating an event through the calendar API. A meeting-preparation agent will gather prior emails, recent calendar history, and meeting notes or transcripts, then generate a concise brief with agenda suggestions, unresolved issues, and likely decisions that need attention. A follow-up-task agent will read the meeting record or resolved email thread, extract commitments and deadlines, and write them into a task system so they can be tracked explicitly. The shared state among these agents will include message IDs, event IDs, task IDs, evidence snippets, confidence scores, and pending approvals.

The methods will combine structured function calling, retrieval, state tracking, and guarded execution. Official Google Workspace APIs provide a practical starting point because Gmail supports messages, threads, search, labels, and drafts; Google Calendar supports event creation and modification; and Google Tasks supports lightweight follow-up tracking. The proposal will keep Microsoft Graph mail and calendar support as a comparison point or stretch goal rather than a core requirement, because supporting multiple enterprise suites could overexpand the scope. On the model side, the team will use a primary LLM with strong function-calling and long-context capabilities for the supervisor and generation-heavy tasks, and may use a smaller model for extraction-oriented subtasks if time allows. The important methodological claim is not a particular vendor choice, but the layered design: planning, retrieval, tool execution, and error recovery are separated so that failures can be diagnosed rather than treated as generic agent mistakes.

## Data, resources, and recent related work

The project will use a hybrid data strategy built around real APIs over synthetic or sandboxed office data. This choice is important because private organizational email and calendar data are difficult to access and not reproducible for class evaluation. The main environment will therefore be one or more synthetic Google Workspace test accounts populated with seeded inboxes, meetings, and task states. This approach is directly inspired by WorkBench and OfficeBench, both of which argue for realistic but controlled workplace scenarios. By seeding accounts with recurring meetings, time-zone constraints, unfinished tasks, and multistep email threads, the team can create an auditable evaluation set with clear expected outcomes.

Public datasets will support the content layer of the benchmark. The Enron Email Dataset provides realistic email language and thread structure for inbox triage and response drafting, although the proposal should acknowledge that it is old and will require relabeling or synthetic task construction. For meeting understanding, the AMI Meeting Corpus and ICSI Meeting Corpus offer meeting transcripts and annotations, and QMSum is especially useful because it evaluates query-focused meeting summarization, which aligns well with a secretary task such as preparing context for tomorrow’s meeting. MeetingBank offers newer meeting-summary data in a format closer to business minutes and may help with follow-up generation. These datasets will not by themselves define the entire benchmark; instead, they will help populate realistic email and meeting content inside the API-backed sandbox.

Recent related scientific writing makes the project both concrete and current. WorkBench is the closest benchmark precedent because it evaluates agents in a realistic workplace setting with many tools and outcome-centric scoring. OfficeBench is directly relevant because it centers office automation across email, documents, and calendar tasks. OSWorld broadens the motivation by showing that multimodal computer-use remains challenging in realistic environments, which matters if the project later extends beyond direct APIs to browser-based interaction. NATURAL PLAN provides a useful planning-focused comparison because it includes meeting-planning and calendar-scheduling tasks that can help distinguish planning quality from API execution quality. API-Bank and the Berkeley Function Calling Leaderboard are useful lower-level references for tool-calling reliability. Finally, the OpenClaw repository and official documentation matter not as scientific evidence of benchmark performance, but as the authoritative description of the platform the project intends to build on.

## Planned methods and multi-agent workflow

The planned workflow begins with a trigger, either a user instruction or a new incoming email detected in the sandbox inbox. The supervisor agent first classifies the requested outcome and determines whether the task is local to one application or spans several. For example, a scheduling email may require reading the message, checking the calendar, comparing constraints, and drafting a response. The supervisor then invokes the inbox-triage agent to interpret the email and extract the relevant entities, such as participants, deadlines, urgency, and whether a meeting is being requested. If the task requires coordination, the calendar agent queries the calendar API, checks available time slots, resolves hard constraints such as time zones or mandatory attendees, and returns a set of feasible meeting plans. The supervisor can then ask the triage agent to draft a reply or directly prepare an event proposal.

When a meeting is approaching, the meeting-preparation agent retrieves related email threads, prior meeting notes, outstanding tasks, and any relevant summaries, then generates a short prep brief for the user. The brief will emphasize decisions already made, unresolved questions, likely blockers, and action items that should be revisited. After the meeting or after a thread is resolved, the follow-up-task agent extracts explicit and implicit commitments, converts them into structured tasks with owners and tentative deadlines, and records them in Google Tasks. Throughout the workflow, the supervisor keeps a lightweight workspace memory that stores state references and provenance so that later steps can be traced back to earlier evidence. This memory is also important for approval and recovery. If the system is uncertain or a requested action is high risk, it should ask for confirmation before sending an email or changing a meeting.

This multi-agent workflow is intentionally more concrete than a generic “planner plus tools” description. It gives the project a clear engineering target and a clean comparison against baselines. The most important baseline will be a single-agent system that uses the same underlying model and APIs but handles all work in one loop without role specialization or shared state structure. A second baseline will combine simple rules with an LLM, for example using heuristics for inbox filtering and scheduling constraints while leaving drafting and summarization to the model. These comparisons will allow the team to test whether OpenClaw’s multi-agent organization adds value beyond ordinary tool calling.

## Evaluation plan

Success will be measured primarily through execution-based, outcome-centric evaluation on realistic office scenarios. This follows the strongest precedent in the research literature, especially WorkBench, because office workflows often admit multiple valid action sequences while still having a clear correct end state. The primary metric will therefore be end-to-end task success rate: whether the final inbox, calendar, and task states match the intended result. Example tasks will include triaging a mixed inbox, coordinating a meeting across participants with constraints, producing a useful prep brief for a scheduled meeting, and converting a meeting record into explicit follow-up tasks. Integrated scenarios will also be included, such as reading an email that requests a meeting, checking the calendar, drafting the reply, scheduling the event, and recording the follow-up.

Several secondary metrics will capture quality and safety. For inbox triage, the system will be evaluated using precision and recall for urgency and intent labels. For calendar coordination, the project will report scheduling feasibility rate, defined as the fraction of proposed meetings that satisfy hard constraints such as availability, duration, time zone, and required attendees. For follow-up extraction, the team will compute precision, recall, or F1 against a small annotated gold set of meeting or thread-derived action items. For meeting preparation, the project will use either human ratings or benchmark-aligned criteria to score factuality and usefulness. Operational metrics will include latency, number of tool calls, and user oversight burden, such as how many approvals or manual edits are required before execution. Because secretary work is high risk, the project will also report a critical error rate covering wrong-recipient drafts, incorrect meeting times, dropped tasks, or duplicated events.

The evaluation will include both component-level and end-to-end analysis. NATURAL PLAN can provide a planning-only reference for meeting and calendar reasoning, while API-Bank or BFCL can help diagnose whether failures are due to weak tool invocation rather than poor high-level planning. The main reported comparisons, however, will be the end-to-end multi-agent system versus the single-agent baseline and the rules-plus-LLM baseline. A strong result for the project would not require full autonomy. It would be sufficient to show that a constrained OpenClaw secretary improves end-to-end workflow completion and reduces user work on realistic tasks without increasing critical errors.

## Risks and open questions

The biggest project risk is the tension between realism and privacy. Real office data would make the project more convincing, but it is not reproducible and raises obvious privacy concerns. The proposal therefore centers evaluation on synthetic or sandboxed accounts, with any human pilot treated as optional and purely formative. A second risk is benchmark mismatch. Public datasets cover email, meetings, planning, or tool use separately, but none perfectly capture the full secretary workflow. The hybrid benchmark design is intended to bridge that gap, but it will require careful task construction. A third risk is scope. Supporting multiple productivity suites, rich long-term memory, and full autonomy may be too ambitious for one course project, so the initial build should focus on Google Workspace APIs, bounded task types, and human confirmation for high-risk actions.

There are also several open questions worth naming explicitly. One is whether the available OpenClaw setup already exposes the exact connectors needed for Gmail, Calendar, and Tasks in the class environment, or whether the team will need to implement some wrappers itself. Another is how much annotation effort is feasible for a gold set of triage labels and action items. A third is how meeting-preparation quality should be scored, whether by direct human judgment, benchmark overlap, or both. These open questions do not weaken the proposal; they clarify where the project may need to narrow scope or make principled design choices.
