# Context For Thorough Single-Thread LangSmith / LangGraph Error Analysis

We are analyzing LangSmith/LangGraph traces for a TravelPlanner project.

There are two systems:

1. `baseline/`
   - Single-agent baseline.
   - Uses only Tavily web search.
   - Local JSON exports contain `outputs.messages`.
   - Baseline messages include real `tool_calls`, e.g. `tavily_web_search`.

2. `travel_agent/`
   - Multi-agent TravelPlanner system.
   - Described in `paper.tex`.
   - Uses planner, execution agent, domain subagents, TravelPlan mutation tools, and validator/reviewer.
   - Local JSON exports contain final artifacts:
     - `outputs.query`
     - `outputs.task_list`
     - `outputs.travelplan`
     - `outputs.validation_passed`
     - `outputs.validation_attempts`
     - `outputs.message_histories`
   - Local JSON exports do not contain full execution-agent raw tool-call history.
   - Full LangSmith data was exported separately.

## Recommended Single-Thread Workflow

Given a root run id or thread id:

1. Find the row in:

```text
thread_analysis/travel_agent_full/manifest.csv
```

or:

```text
thread_analysis/travel_agent/manifest.csv
```

2. Filter all relevant files by `thread_id`.

Useful Pandas pattern:

```python
manifest = pd.read_csv("thread_analysis/travel_agent_full/manifest.csv")
runs = pd.read_csv("thread_analysis/travel_agent_full/langsmith_runs.csv")
messages = pd.read_csv("thread_analysis/travel_agent_full/messages.csv")
tool_calls = pd.read_csv("thread_analysis/travel_agent_full/tool_calls.csv")

thread_id = manifest.loc[manifest.root_run_id.str.contains("019e31a3"), "thread_id"].iloc[0]

runs_t = runs[runs.thread_id == thread_id]
messages_t = messages[messages.thread_id == thread_id]
tool_calls_t = tool_calls[tool_calls.thread_id == thread_id]
```

For real executed tools only, prefer:

```python
runs = pd.read_csv("thread_analysis/travel_agent/langsmith_runs.csv")
runs_t = runs[runs.thread_id == thread_id]
```

## Error Categories From `paper.tex`

Paper taxonomy:

```text
T1: Invalid Arguments
T2: Missing Mandatory Parameters
T3: Redundant / Infinite Loops
T4: Misinterpreting Tool Outputs
T5: Link Rot / Empty Crawl
T6: Cross-Agent Data Corruption
```

Current operational definitions:

### Repeated Tool Calls

For actual tool-run export:

```python
groupby(["thread_id", "name", "args_norm"]).size() > 1
```

This detects the exact same tool with identical normalized args.

### Repeated Queries

For search/domain tools only:

```text
search_web
search_restaurants
search_attractions
search_flights
search_hotels
check_route_timing
build_place_distance_graph
```

Normalize query/request-like fields from `inputs_json`.

Repeated query definition:

```python
groupby(["thread_id", "name", "query_norm"]).size() > 1
```

### Dead Loops

Current indicators:

1. Consecutive exact repeat:
   - same tool
   - same args
   - adjacent in call order

2. Same call 3+ times:
   - same tool
   - same normalized args
   - same thread

3. Mutation backtracking loop:
   - patterns like:
     - `add_slot -> delete_slot -> add_slot -> delete_slot`
     - `insert_slot -> delete_slot -> insert_slot -> delete_slot`
     - `delete_slot -> add_slot -> delete_slot -> add_slot`

This is most relevant for the TravelPlanner execution agent because it mutates a shared TravelPlan.

### Timeouts / Long-Running Tool Calls

No explicit tool errors were found in the clean `run_type=tool` export.

Timeout risk was approximated by:

- p99 latency outliers
- output text containing:
  - `timeout`
  - `timed out`
  - `failed`
  - `error`
  - `exception`
  - `rate limit`
  - `unavailable`
  - `could not`
  - `unable`

### Cascading Errors

Indicators:

1. `validation_attempts > 1`
   - validator repair loop

2. Weak evidence followed by plan mutation:
   - a tool output has uncertainty/error language
   - later calls include `add_slot` or `insert_slot`

3. Delete after prior insertion:
   - `delete_slot` after prior `add_slot` or `insert_slot`
   - interpreted as repair/backtracking

Suggested paper interpretation:

The corrected LangSmith export contains 2,300 real TravelPlanner tool runs across 15 LangGraph threads. Grouping tools by architectural role shows that most error indicators concentrate in the TravelPlan mutation layer rather than in a single domain-search agent. This is expected for the multi-agent design: subagents retrieve evidence, but the execution agent repeatedly mutates the shared TravelPlan object while repairing budget, timing, or validation issues. We therefore interpret repeated add/delete or insert/delete patterns as repair-loop indicators rather than simple duplicate search calls. Domain/search tools still contribute repeated-query and uncertainty signals, especially when retrieved evidence is weak or later embedded into final slots. Structural validation of the final TravelPlan adds a second view of error propagation: missing or fragile links and uncertainty-bearing slot notes reveal cases where retrieval uncertainty survived into the final itinerary artifact.

## Important Caveat

`thread_analysis/travel_agent_full/tool_calls.csv` is huge and contains recursive extraction from message snapshots. It is useful for reconstructing conversation/state, but it overcounts actual tool calls because LangGraph state snapshots duplicate prior messages.

For actual tool execution counts, use:

```text
thread_analysis/travel_agent/langsmith_runs.csv
```

where `run_type == tool`, or the corresponding clean `tool_calls.csv`.

For thorough single-thread reconstruction, use the full export, but deduplicate by:

```python
tool_call_id
```

or use real `run_type=tool` runs as the ground truth.
